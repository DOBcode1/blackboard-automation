## Known small issues (parked, not worth fixing yet)
- Attaching a file then switching chats mid-read loses the in-progress attachment (narrow window; the file still completes server-side).
- Three scraper bugs (Social Psych popover, British Gov lazy-load, International Internship DOM) — need a second Blackboard account to test.
- Haiku router JSON fragility — route_question_to_courses parses Haiku's output with a bare json.loads. Observed a live JSONDecodeError fall back to all courses (fail-open worked, user got a correct answer, but the fallback sent ~32K tokens to Sonnet at ~$0.11 for one query). Likely cause: Haiku occasionally wrapping its JSON array in markdown ```json fences. Low-effort fix (strip code fences before parsing) but belongs in P2's extraction-robustness work, not a one-off patch. Flagged because the silent failure has a real cost tail — every fail-open query costs like a worst-case all-courses query, which is the exact scenario the router exists to prevent.
- Vision OCR of publisher/scanned PDFs is blocked by Anthropic's content filter (confirmed live: "Output blocked by content filtering policy" on a scanned Ethics PDF). Code handles it cleanly — logs status=failed, extraction_method=vision_ocr — but the USER-FACING story is bad: the user waits through polling and gets a silent failure with no explanation. Entangled with the Stage C attorney hold. P4 should add a clear user-facing message for filter-blocked uploads. Not a code bug.
- Measured per-query cost is at/above the README's $0.05-0.10 target. First logged runs: a normal 6-course chat = ~$0.099 (32K input tokens); an attachment-summary chat = ~$0.127 (34K tokens). Router was working correctly (all-courses is right for general questions) — the cost driver is injecting full course summaries as context, not a routing failure. This is direct measured evidence that Stage D (move summaries into retrieval rather than inject whole) and the Phase 10 structured-routing short-circuit are unit-economics necessities, not just cleanups. Captured here so the dollar figures aren't lost before that work happens.
- Friday-default bug in query.py (CONFIRMED, root cause known) — The preprocessing resolver defaults day-unspecified "Week N" references to the FRIDAY of that week, overriding weekday information that is actually present in the source text. Two confirmed cases: (a) Fintech assignment whose raw text says "Week 10 (Tuesday seminar session)" — chat correctly answers Tuesday March 17, but the calendar places it Friday March 20; (b) Legal final, which Blackboard has set to Thursday, shows on the calendar as Friday. The information is not lost — the day is in the prose — the resolver rule just ignores it. Fix: change the resolver rule to read the weekday from the text (or a known class schedule) and only leave the day uncertain / lower confidence when no day is available. Must NOT alter dates that came from an explicit Blackboard due date or explicit weekday. Likely a single prompt change in query.py — confirm no second resolution site exists first.
- Preprocessing non-determinism (observed, likely benign) — Running the pipeline on a different machine produced different routing of ungraded items than the current working data. The newer run correctly routed ungraded discussion posts, participation, and extra-credit items to NEEDS-ATTENTION; the current working data incorrectly placed those same discussion posts directly on the calendar. Both runs captured the same items — this is classification/date-resolution differing across runs, most likely because preprocessing calls Sonnet with an unpinned temperature. Note: the newer run's behavior is the CORRECT one. Cheap mitigation to test: pin preprocessing temperature low. Caveat: runs were across different code (pre/post chunking) and machines, so not a clean A/B — the regression check in Immediately Next will serve as the controlled same-machine observation. Update: a controlled-ish re-run on the SAME machine/codebase reproduced the ungraded-item misclassification and was slightly WORSE — it placed discussion posts AND participation items on the calendar as if graded deadlines, where the correct behavior is routing them to needs-attention. This is the graded-vs-ungraded classification swinging across runs (separate from weekday resolution, which the Friday fix corrected and which held correctly this run: Fintech presentation resolved to Tuesday, real Friday finals preserved). Strengthens the case that preprocessing temperature is unpinned and should be set low. Caveat: one re-run is not a clean A/B — the real test is whether pinning temperature makes the classification STABILIZE (consistent across runs) and ideally stabilize on the correct outcome. RESOLVED CAUSE: The instability is NOT temperature. A controlled 3-run test at temperature=0.0 still produced swinging deadline counts (56/55/58 total, resolved 41/37/38, needs-attention 15/18/20) and flickering discussion-post classification. Root cause confirmed via cost logs: the preprocessing stage (preprocess_courses) is a WHOLE-COURSE single LLM call, and Legal Framework (195 items, ~68K input tokens) hits the max_tokens=4096 OUTPUT ceiling — it is the only course that pins at exactly 4096; all others finish at 2,655–3,321. When output is truncated mid-generation, WHICH assignments survive before the cutoff varies slightly run-to-run, destabilizing the deadline set. The temperature pin was kept (correct for extraction; quiets the small residual jitter seen on smaller courses) but did NOT and could not fix the truncation-driven wobble. Near-term fix: raise preprocessing max_tokens so large courses finish — must verify the course completes BELOW the new ceiling and counts stabilize, not just that the number is higher. OUTCOME (resolved): max_tokens raised 4096->8192 — verified Legal Framework now finishes at out=5034, no longer truncating. This fixed the truncation-driven item dropout but did NOT fully stabilize the calendar; the remaining instability was a separate downstream routing issue (see next entry). Both fixes committed.
- Preprocessing/extraction scaling (connected finding) — The truncation bug, the high preprocessing cost (Legal/Ethics/Fintech preprocessing calls run $0.21–0.55 each, partly from large whole-course context), and the existing Stage D plan are the SAME root issue: the deadline-EXTRACTION stage still processes whole courses in single LLM calls and does not scale. Note: the Stage A chunking migration chunked the CHAT/retrieval path (build_context), but NOT the preprocessing/extraction path — that stage was never chunked and still injects whole-course key-item text in one call. Raising max_tokens is a symptom fix; the structural fix is Stage D (move summaries into retrieval) and likely chunking the preprocessing INPUT for very large courses. Captured so the connection isn't lost.
- Aggregator desync after re-preprocessing — Regenerating the preprocessing cache does NOT automatically rebuild output/deadlines.json. On app startup the server LOADS the existing deadlines.json rather than re-aggregating, so after a fresh preprocessing run the calendar silently shows stale aggregated data until `python aggregate_deadlines.py` is run manually. Confirmed live: a re-run showed identical calendar output until the aggregator was run by hand. Operational/silent-staleness risk — a user (or future-self) who re-scrapes would trust a calendar built from stale deadlines. Not fixing now; recording as a P2/operational finding.
- Calendar bucketing stabilized (discussion flicker + confidence cliff) — The flickering of ungraded discussion posts on/off the calendar was NOT a classification rule in the preprocessing prompt (that prompt has no calendar-vs-needs-attention concept). The routing decision lives in aggregate_deadlines.py: an item is calendar-resolved when due_resolved is non-null AND effective_confidence >= 3, with NO filtering by assignment type. Discussions flickered because borderline ungraded items wobble across the date/confidence threshold run-to-run.
  - FIX (committed, verified across 2 runs): (a) type-scoped threshold — items typed 'discussion' or 'participation' now require confidence >= 4 to reach the calendar; all other types unchanged at >= 3. Confirmed graded discussions with real Blackboard dates (conf 4-5) still pass, so legitimate graded discussions are not buried. (b) _parse_confidence default changed 3->2 so items with missing/unparseable confidence route to needs-attention instead of silently clearing the calendar bar. RESULT: Legal discussions now stay off-calendar consistently; total deadline count stabilized at 57 (was swinging 59/56/56). Residual one-item wobble (resolved 30 vs 31) is irreducible LLM non-determinism, not the discussion problem.
- Confidence-3 cliff — legitimate dated items can flicker — Discovered while protecting two real items (Personal Learning Statement [presentation], April 8 Written Summary [homework]) during the bucketing fix. Both sit at confidence 3 with vaguely-worded but real dates (e.g. 'April 8, Wednesday 6-8pm'). Confidence 3 is exactly the calendar threshold, so when the model self-scores such an item a 2 on a given run, it drops off the calendar. This is upstream model wobble, not an aggregator bug — no threshold change can distinguish a real conf-3 item from conf-3 noise at bucketing time.
  - Two-layer fix: (1) For the user now: confirm-and-save via the edit modal sets effective_confidence=5 (line 207-208), pinning the item on-calendar permanently, immune to re-preprocessing. The override system is the designed home for 'vague date I've verified.' (2) For the product (root-cause, prompt-level): added an EXPLICIT-DATE FLOOR to the preprocessing confidence rubric — any item with an explicit calendar date in the source text (even embedded in descriptive text) scores confidence >= 4, never 1 or 2. This makes legitimately-dated items clear the calendar bar every run instead of flickering. COMMITTED but NOT YET VERIFIED — verification deferred to the next genuine re-scrape to avoid a paid test run.
  - PRODUCT RISK noted: a beta user won't know to manually lock a real deadline that silently failed to appear on a given run. The explicit-date floor is the user-facing mitigation; the override system only protects users who already know what's missing.
- Test-cost discipline — Full six-course re-preprocessing costs ~$1.70/run (Philosophical Ethics ~$0.55/167K tokens, Fintech ~$0.47/141K — large whole-course contexts). Multiple verification loops this session added up. GOING FORWARD: do not re-preprocess solely to test a change. Verify preprocessing-prompt changes opportunistically on the next real re-scrape, or build a small single-course / cached fixture for cheap iteration. Reinforces Stage D urgency (whole-course single-call extraction is the cost driver).
- PARKED: P2 adversarial testing (return to this before/during micro-beta) — Deferred by choice to move on to P3/new feature work. NOT abandoned — return when ready. The calendar-stability half of P2 is done (truncation fix, discussion-flicker fix, explicit-date floor). What remains is the adversarial corpus to catch CONFIDENTLY-WRONG failures (worse than missing items, because users trust a displayed deadline).
  - Trimmed scope when picked back up (full 14-case harness NOT needed — ~half the original cases already pass on real data and were demoted to regression checks). Focus on the genuine unknowns:

## Key decisions & principles (orient fast)
- The scraper is PLUMBING, not the moat. Moats = distribution, per-school data network effects, workflow lock-in. Everything from ingestion.py onward is already REST-API-ready; migrating off Playwright means writing a new ingestion script feeding the same document store — the coupling is the data SCHEMA, not Playwright. Don't over-invest in the scraper; don't parallelize it before the API-path decision.
- AI is invisible plumbing — positioned as removing student cognitive load, NOT as an "AI study tool."
- Cost discipline: cheap router (Haiku) + expensive generator (Sonnet); fail-open routing (errors fall back to all courses). RAG exists so a heavy user can't cost more than the subscription.
- Legal: reproducing publisher EXPRESSION is the exposed area; facts (deadlines, schedules) and the student's OWN content are safe. Pursue institutional sanction — do not engineer around content filters or hide infringement.
